## 1. UX Audit — Messages d'erreur compréhensibles

### Analyse de `formatEdgeFunctionError()` (new.tsx:133-161)

L'objectif de la story est de remplacer le message générique "Consultez les logs Supabase → Functions → create-user-profile" par des messages lisibles pour un admin non-technique.

| Cas Edge Function | Message affiché | Compréhensible ? |
|-------------------|-----------------|------------------|
| `[step: env-check]` | "Erreur de configuration serveur : variable SUPABASE_SERVICE_ROLE_KEY manquante. Contactez l'administrateur technique." | ✅ Clair, actionnable |
| `[step: auth-create]` / `[step: auth-invite]` | "Erreur lors de la création du compte : {message nettoyé}" | ✅ Clair, indique l'étape |
| `[step: profile-insert]` / "Profile creation failed" | "Erreur lors de l'enregistrement du profil : {message nettoyé}" | ✅ Clair, indique l'étape |
| `[step: role-check]` / "Forbidden" | "Erreur d'autorisation : votre compte n'a pas le rôle admin requis. Reconnectez-vous et réessayez." | ✅ Clair, actionnable |
| `[step: auth-header]` / "Unauthorized" | "Session expirée ou non reconnue. Veuillez vous reconnecter." | ✅ Clair, actionnable |
| `[step: body-validate]` / `[step: body-parse]` | "Erreur de configuration serveur. Contactez l'administrateur technique." | ✅ Acceptable |
| Fallback `[step: xxx]` inconnu | "Erreur technique : {rawMessage}" | ⚠️ Affiche le raw message technique — acceptable comme dernier recours |
| `non-2xx` sans step | "La fonction de création a retourné une erreur non identifiable. Consultez les logs Supabase → Functions → create-user-profile." | ⚠️ Renvoie aux logs — acceptable uniquement si le step n'a pas pu être parsé |

**Verdict UX** : Les messages sont globalement compréhensibles pour un admin. Le fallback final renvoyant aux logs Supabase reste acceptable car il ne se déclenchera plus dans le cas normal (le `.clone()` permet maintenant d'extraire le `step`). L'amélioration est substantielle par rapport à l'état précédent.

## 2. Bug Hunting — Edge Cases

### E1 — Edge Function ne retourne pas de JSON valide

**Scénario** : L'Edge Function crashe avant d'écrire une réponse (ex. : timeout, OOM), ou retourne du HTML (page d'erreur proxy).

**Flux** :
1. `supabase.functions.invoke()` lève `FunctionsHttpError`
2. `extractFunctionError()` tente `response.clone().json()`
3. `json()` lève une exception (body HTML non parseable)
4. Le `catch {}` attrape silencieusement → fallback vers `baseMsg` (message SDK)
5. `formatEdgeFunctionError(baseMsg)` → branche `non-2xx` → message générique renvoyant aux logs

**Résultat** : Message générique affiché, non bloquant pour l'utilisateur (page reste utilisable), mais l'admin doit consulter les logs.

**Verdict** : Comportement acceptable — c'est précisément le cas où on ne peut pas faire mieux sans connaître l'erreur réelle. Le `.clone()` implémenté résout le cas principal (JSON parseable mais stream consommé).

---

### E2 — Champ `step` absent du body JSON

**Scénario** : L'Edge Function retourne `{ "error": "Some error message" }` sans champ `step`.

**Flux** :
1. `body.step` est `undefined`
2. `stepSuffix = ''` (pas de suffix)
3. `return new Error(body.error)` — message sans step
4. `formatEdgeFunctionError(rawMsg)` → aucun `[step:]` → tombe dans `return rawMessage`
5. Le message brut de l'Edge Function est affiché tel quel

**Résultat** : Le message de l'Edge Function est affiché directement à l'utilisateur. Si ce message est technique (ex. "duplicate key value violates unique constraint"), il sera peu lisible.

**Verdict** : WARNING — cas acceptable pour l'instant car les versions récentes de l'Edge Function injectent systématiquement `step`. Les anciennes versions (sans `step`) afficheront le message brut, ce qui reste mieux que le message générique précédent.

---

### E3 — Coupure réseau pendant la soumission

**Scénario** : L'utilisateur perd sa connexion entre le clic sur "Créer la fiche" et la réponse.

**Flux** :
1. `handleSubmit()` appelle `createProfileFiche()`
2. `supabase.functions.invoke()` lève une exception réseau (ex. `TypeError: Failed to fetch`)
3. Le `catch (e: unknown)` du `handleSubmit` attrape l'exception
4. `rawMsg = e.message ?? 'Erreur inattendue.'`
5. `formatEdgeFunctionError(rawMsg)` → aucun des cas ne matche `Failed to fetch` → `return rawMessage`
6. Le message "Failed to fetch" ou "NetworkError" est affiché dans `errorBanner`

**Résultat** : Message technique "Failed to fetch" affiché à l'utilisateur, sans indication d'action.

**Verdict** : WARNING — le message n'est pas user-friendly pour ce cas spécifique. Amélioration suggérée dans une story de polish.

---

### E4 — `step` présent mais valeur inattendue (ex. step inconnu futur)

**Scénario** : Une future version de l'Edge Function ajoute un step `[step: new-step]` non géré.

**Flux** :
1. `formatEdgeFunctionError()` ne matche aucun cas spécifique
2. Matche le fallback `rawMessage.includes('[step:')` → `return \`Erreur technique : ${rawMessage}\``
3. Message affiché : "Erreur technique : {message} [step: new-step]"

**Résultat** : Affichage semi-technique mais explicite — l'admin voit l'étape précise. Acceptable.

---

### E5 — `tenantId` absent (new.tsx:165)

**Scénario** : L'auth store ne retourne pas de `tenantId` (session corrompue).

**Flux** :
1. `if (!tenantId)` → retour immédiat avec `setResult({ ok: false, msg: 'Tenant non défini.' })`
2. Message affiché dans `errorBanner` : "Tenant non défini."

**Résultat** : Message technique peu compréhensible pour un admin non-technique.

**Verdict** : WARNING — message "Tenant non défini" est trop technique. Suggéré : "Erreur de session : veuillez vous reconnecter."

---

### E6 — Bouton "Créer la fiche" cliqué deux fois rapidement

**Scénario** : L'utilisateur double-clique le bouton.

**Flux** :
1. Premier clic : `setSubmitting(true)`, bouton `disabled={submitting}` → bouton immédiatement désactivé
2. Deuxième clic : ignoré car `disabled`

**Résultat** : Protection correcte contre la double-soumission. ✅

---

### E7 — `response` déjà consommée avant `clone()` (cas principal du bug)

**Scénario** : Le SDK Supabase lit le body avant d'exposer `error.context`.

**Fix implémenté** : `.clone()` est appelé avant `.json()`. Si le clone échoue (ex. réponse déjà close), l'exception est attrapée et le fallback SDK est utilisé.

**Résultat** : Résout le bug principal décrit dans la story. ✅

## 3. Design Critic — Conformité tokens après corrections Gate 1

Suite aux corrections Gate 1, toutes les couleurs hardcodées du fichier `new.tsx` ont été remplacées par des tokens `@aureak/theme`. Vérification des tokens non conformes restants :

| Fichier:ligne | Valeur résiduelle | Justification |
|---------------|-------------------|---------------|
| new.tsx:stepDotActive | `rgba(193,172,92,0.15)` | Variante transparente de `colors.accent.gold` — pas de token dédié pour 15% opacity, acceptable |
| new.tsx:errorBanner | `rgba(224,82,82,0.12)`, `rgba(224,82,82,0.3)` | Variantes transparentes de `colors.accent.red` — pas de token dédié, acceptable |
| new.tsx:successIcon | `rgba(76,175,80,0.15)` | Variante transparente de `colors.status.present` — acceptable |

Ces rgba sont des variantes d'opacité de tokens existants — l'approche hex+opacity (`` `${colors.accent.gold}1F` ``) ne peut pas être appliquée à des rgba construits (ils utilisent des valeurs RGB séparées). Tolérées comme INFO.
