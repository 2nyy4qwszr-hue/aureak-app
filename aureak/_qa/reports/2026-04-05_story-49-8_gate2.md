## 2. UX Auditor — Clarté fallbacks et états vides

### 2a. Fallback "Données indisponibles" (erreur API)

**Analyse de la card KPI :**

```tsx
sub={statsError ? 'Données indisponibles' : avgAttendance === null ? 'Aucune donnée sur la période' : ...}
accent={statsError ? colors.text.muted : rateColor(avgAttendance)}
```

- Sur erreur : valeur affichée = `'—'`, sous-texte = `'Données indisponibles'`, accent = `colors.text.muted` (gris) → visuel neutralisé, non alarmiste.
- L'utilisateur comprend que la donnée n'est pas disponible sans voir d'erreur technique.

**Analyse du fallback section implantations :**

```tsx
{statsError ? '⚠️' : '📊'}
{statsError ? 'Données indisponibles' : 'Aucune donnée disponible'}
{statsError
  ? 'Impossible de charger les statistiques. Vérifiez votre connexion ou contactez le support.'
  : ...}
```

- Distinction icône + titre + message entre erreur (⚠️) et vide (📊) : claire et perceptible.
- Le message d'erreur propose une action (vérifier connexion / contacter support) : bonnes pratiques UX.

**Verdict : PASS**

### 2b. Distinction erreur vs vide

| Scénario | Icône | Titre | Message | Verdict |
|----------|-------|-------|---------|---------|
| Erreur API (`statsError=true`) | ⚠️ | "Données indisponibles" | "Vérifiez connexion / support" | Clair |
| Vide sur période, filtre global | 📊 | "Aucune donnée disponible" | "Ajustez la période..." | Clair |
| Vide sur implantation filtrée | 📊 | "Aucune donnée disponible" | "Aucune séance pour {nom}..." | Clair et contextualisé |

**Verdict : PASS**

### 2c. KPI cards en état erreur

Les KPIs de présence et maîtrise affichent `'—'` avec sous-texte `'Données indisponibles'` lorsque `statsError=true`. La card Séances affiche `'—'` avec `'Données indisponibles'`. Cohérent.

**NOTE UX** : Les KPIs Joueurs, Coachs, Groupes (`getDashboardKpiCounts`) ont leur propre état de chargement (`loadingCounts`) mais **pas d'état d'erreur distinct** — en cas d'échec, ils affichent `'—'` (valeur nulle) sans différenciation visuelle d'une valeur réellement zéro. L'utilisateur ne peut pas distinguer "0 coachs" de "erreur chargement coachs". **WARNING UX non bloquant** pour cette story.

**Verdict : PASS avec 1 WARNING**

## 3. Bug Hunter — Edge cases et bugs résiduels

### 3a. Division par zéro SQL — PASS

Les deux taux utilisent `NULLIF(..., 0)` :
```sql
ROUND(100.0 * COUNT(a.id) FILTER (...) / NULLIF(COUNT(a.id), 0), 1)
ROUND(100.0 * (...) / NULLIF((...), 0), 1)
```
Division par zéro impossible. Retourne `NULL` si aucune donnée → géré côté frontend par `?? 0` et vérification `!== null`.

**Verdict : PASS**

### 3b. Edge case p_from > p_to (dates inversées)

Si l'utilisateur entre une date "Du" supérieure à "Au" en mode custom, `BETWEEN p_from AND p_to` retourne 0 lignes en SQL (BETWEEN est sensible à l'ordre). Le dashboard affiche un état vide sans erreur — perceptible uniquement comme "Aucune donnée disponible". **Pas de validation frontend** de l'ordre des dates.

**Verdict : WARNING-B1** — pas de BLOCKER (pas de crash, pas de donnée erronée) mais UX confusante. L'utilisateur pourrait penser à un vrai vide de données plutôt qu'à une erreur de saisie.

### 3c. État statsError non propagé depuis le catch global

```tsx
const load = async (f = from, t = to) => {
  setStatsError(false)
  setLoading(true)
  try {
    ...
    if (statsResult.error) { setStatsError(true) }
    ...
  } catch (err) {
    // setStatsError(true) absent ici !
    if (process.env.NODE_ENV !== 'production') console.error('[dashboard] load error:', err)
  } finally {
    setLoading(false)
  }
}
```

En cas d'exception réseau (non-erreur API : timeout, fetch échoué), `statsError` reste `false` (réinitialisé en début de `load`), et `stats` reste au tableau vide initial (ou à sa valeur précédente si rechargement). La section implantations affiche alors "Aucune donnée disponible" (état vide) au lieu de "Données indisponibles" (état erreur).

**Impact** : distinction erreur/vide incorrecte en cas de panne réseau.

**Verdict : WARNING-B2** — non bloquant (pas de crash, pas de donnée erronée, message quand même affiché) mais mismatch UX.

### 3d. Bug logique dans le sous-texte de la card Anomalies (ligne 631)

```tsx
sub={criticalCount > 0
  ? 'critiques'
  : anomalies.length > 0
    ? `dont ${criticalCount} critique`   // ← criticalCount = 0 ici !
    : 'aucune alerte'
}
```

Quand `anomalies.length > 0` ET `criticalCount === 0` : affiche **"dont 0 critique"** — sémantiquement incorrect. Devrait afficher "aucune critique" ou simplement "actives".

**Impact** : texte trompeur visible en production dès qu'il existe des anomalies de type warning/info sans critique.

**Verdict : BLOCKER-B1** — correction requise.

### 3e. Edge case aucune implantation

Si `implantations` est vide (académie non configurée), le sélecteur affiche uniquement "Toutes les implantations", les stats sont vides, le dashboard affiche "Aucune donnée disponible". Comportement acceptable — pas de crash.

**Verdict : PASS**

### 3f. `useEffect` sur `load` sans dépendances

```tsx
useEffect(() => { load() }, [])
```

`load` capture `from` et `to` via les paramètres par défaut (`f = from, t = to`) — les valeurs initiales sont correctes. Les rechargements ultérieurs passent explicitement les nouvelles valeurs. Pas de bug fonctionnel, uniquement une lint warning React (exhaustive-deps).

**Verdict : WARNING non bloquant (déjà signalé Gate 1)**
