## Bug Hunt — Edge Cases

### Thème déjà ajouté

**Scénario** : L'utilisateur ouvre le picker alors qu'un thème est déjà dans la liste.
**Résultat** : ✅ Le filtre `!themeBlocks.map(b => b.themeId).includes(t.id)` exclut correctement les thèmes déjà associés.
**Edge case — double clic rapide** : Si l'utilisateur clique deux fois sur un thème avant que `handleAddTheme` complète, le bouton du picker n'est pas désactivé per-item. `addingTheme` désactive le bouton "+ Ajouter" principal mais la ligne du picker reste cliquable pendant l'ajout.
**Sévérité** : WARNING — la double soumission entraînerait une erreur Supabase (contrainte d'unicité `(session_id, theme_id)`) qui serait capturée par `if (error)` dans `handleAddTheme` et affichée dans `themeAddError`.

### Liste de thèmes vide (bibliothèque vide)

**Scénario** : `listMethodologyThemes({ activeOnly: true })` retourne `[]`.
**Résultat** : ✅ Le picker s'ouvre avec le message "Tous les thèmes actifs sont déjà associés" (impropre si la bibliothèque est vide, mais non bloquant).
**Note** : Le message serait légèrement trompeur — "Tous les thèmes actifs sont déjà associés" quand en réalité il n'y a aucun thème actif. WARNING UX mineur.

### Erreur réseau pendant la suppression

**Scénario** : `removeSessionThemeBlock(blockId)` retourne `{ error: 'network error' }`.
**Résultat** : ✅ Rollback vers `snapshot` + `setThemeRemoveError(error)` (après correction Gate 1). L'utilisateur voit le thème réapparaître avec un message d'erreur explicite.
**Vérification** : `handleRemoveTheme` a `setRemovingThemeId` en `finally` → pas de state bloqué.

### `sessionId` undefined

**Scénario** : `useLocalSearchParams` retourne `sessionId: undefined`.
**Résultat** : ✅ Guard `if (!sessionId)` en tête de `load()` (ligne 105) + `if (!session || !sessionId) return` dans `handleAddTheme` (ligne 210). Robuste.

### `session.tenantId` undefined

**Scénario** : `session` est null ou `tenantId` manquant.
**Résultat** : ✅ Guard `if (!session || !sessionId) return` protège `handleAddTheme`. Le `session` est non-null à ce stade du cycle de vie (la section `!loading && (...)` garantit que `session` existe).

### Appel API `addSessionThemeBlock` — type de retour `error`

**Vérification** : `error: string | null` dans `addSessionThemeBlock` (confirmé dans `session-theme-blocks.ts` ligne 74). `setThemeAddError(error ?? 'Erreur inconnue')` est correct — pas de cast abusif, pas d'objet Error sérialisé en `[object Object]`.

### Appel `removeSessionThemeBlock` — type de retour `error`

**Vérification** : `error: string | null` (ligne 112 de `session-theme-blocks.ts`). `setThemeRemoveError(error)` est correct.

### `availableThemes` cache — rechargement si bibliothèque change

**Scénario** : L'utilisateur ouvre le picker, ajoute un thème, ferme, rouvre. Le check `if (availableThemes.length === 0)` empêche le rechargement → liste obsolète si un thème a été ajouté à la bibliothèque entre-temps.
**Sévérité** : INFO — comportement acceptable pour UX (évite latence). La liste est rafraîchie à chaque rechargement de page.

### Race condition — fermeture page pendant ajout/suppression

**Scénario** : L'utilisateur navigue avant que l'API réponde.
**Résultat** : WARNING — pas de `useRef(mounted)` ou `AbortController`. Si la page se démonte pendant `handleAddTheme`, `setThemeBlocks(prev => ...)` sera appelé sur un composant démonté → warning React (pas de crash). Pre-existing pattern dans ce fichier.
