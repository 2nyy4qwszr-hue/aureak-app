## UX Auditor

### BLOCKERs UX

1. **Couleur hardcodée** — grep détecte `fontSize` hardcodés et valeurs numériques de spacing dans les styles, mais toutes les couleurs viennent de `tokens.colors.*` ou `methodologyMethodColors`. Pas de hex literals directs. → **OK** (valeurs numériques = WARNING, pas BLOCKER selon historique projet)

2. **État de chargement** — `loading` state → affiche "Chargement…" text. Pas de skeleton. → WARNING (pas BLOCKER — pattern identique au reste du projet)

3. **État d'erreur** — `loadSessions` et `loadExercises` utilisent `try/finally` mais **sans `catch`**. Si l'appel API échoue, l'erreur est silencieuse, aucun feedback utilisateur, la liste reste vide. → **WARNING** (pattern similaire détecté dans stories précédentes, noté non-bloquant)

4. **Composant UI hors @aureak/ui** — `EntraînementsTable` et `ExercicesTable` sont des composants locaux. Justifiable — tables complexes non couvertes par `@aureak/ui`. → **OK**

5. **Route sans index.tsx** — La route est dans `index.tsx` directement (pattern direct valide). → **OK**

## Bug Hunter

### 5 points BLOCKER

1. `.data.field` sans optional chaining → API via `@aureak/api-client`, pas d'accès direct Supabase. → **OK**
2. Double-soumission → pas de formulaire de soumission ici. Delete btn = `disabled={isDeleting}`. → **OK**
3. try/catch silencieux → `loadSessions`/`loadExercises` ont try/finally **sans catch** → **WARNING** — en cas d'erreur réseau, l'array reste vide, aucun feedback. Pattern cohérent avec reste du projet (non-BLOCKER dans historique QA).
4. Erreur Supabase ignorée → les API functions dans `@aureak/api-client` gèrent les erreurs. → **OK**
5. Mutation sans invalidation → pas de TanStack Query utilisé ici (state local useState). → **OK**
