## 2. Findings — Migration 00117

### [BLOCKER] B1 — Policies sans `DO $$ BEGIN ... EXCEPTION WHEN duplicate_object THEN NULL END $$`

**Fichier** : `supabase/migrations/00117_implantations_photo_url.sql` (avant correction)

**Problème** : Les trois `CREATE POLICY` étaient écrits sans wrapping idempotent. Si la migration est rejouée (rollback/push local, branch Supabase), elle échoue avec `ERROR: policy already exists`. Les migrations doivent être idempotentes.

**Pattern de référence** : migrations 00084 (`club_logos_upload_admin`) et 00067 (`child_photos_admin_insert`) utilisent systématiquement le `DO $$ BEGIN ... EXCEPTION WHEN duplicate_object THEN NULL; END $$` pattern.

**Correction appliquée** : Les 3 politiques (+ la nouvelle politique UPDATE) sont maintenant enveloppées dans des blocs DO idempotents.

**Statut** : CORRIGÉ

---

### [BLOCKER] B2 — Policies d'écriture utilisaient `auth.role() = 'authenticated'` au lieu de `(auth.jwt() ->> 'user_role') = 'admin'`

**Fichier** : `supabase/migrations/00117_implantations_photo_url.sql` (avant correction)

**Problème** : La condition `auth.role() = 'authenticated'` autorise **tout utilisateur authentifié** (coach, parent, child, club) à uploader ou supprimer des photos d'implantation. Seuls les `admin` doivent avoir ce droit.

**Exemple de violation** :
```sql
-- AVANT (dangereux)
WITH CHECK (
  bucket_id = 'implantation-photos'
  AND auth.role() = 'authenticated'  -- ← tout user authentifié peut uploader
)
```

**Pattern de référence** :
- Migration 00067 : `AND (auth.jwt() ->> 'role') = 'admin'`
- Migration 00084 : `AND (auth.jwt() ->> 'user_role') = 'admin'`

**Correction appliquée** : Les policies INSERT, UPDATE, DELETE utilisent maintenant `(auth.jwt() ->> 'user_role') = 'admin'`, cohérent avec le pattern 00084 (même claim JWT que le projet Aureak utilise).

**Statut** : CORRIGÉ

---

### [BLOCKER] B3 — Policy UPDATE manquante pour `upsert: true`

**Fichier** : `supabase/migrations/00117_implantations_photo_url.sql` (avant correction)

**Problème** : `uploadImplantationPhoto` utilise `supabase.storage.upload(path, file, { upsert: true })`. L'option `upsert: true` émet une requête INSERT qui se transforme en UPDATE si l'objet existe déjà. Sans policy UPDATE sur `storage.objects`, le ré-upload d'une photo existante échoue silencieusement (erreur Supabase Storage RLS) bien que la policy INSERT soit satisfaite.

**Correction appliquée** : Ajout de `implantation_photos_admin_update` policy pour le bucket, réservée aux admins, dans un bloc DO idempotent.

**Statut** : CORRIGÉ

---

### [BLOCKER] B4 — Bucket sans `file_size_limit` ni `allowed_mime_types`

**Fichier** : `supabase/migrations/00117_implantations_photo_url.sql` (avant correction)

**Problème** : Le bucket `implantation-photos` était créé sans contrainte de taille de fichier ni restriction de type MIME. Un utilisateur admin pourrait uploader n'importe quel type de fichier (exécutables, vidéos lourdes…), contre-indiqué pour un bucket de photos/logos.

**Pattern de référence** : Migration 00067 définit `file_size_limit: 5242880` (5 MB) et `allowed_mime_types: ARRAY['image/jpeg', 'image/png', 'image/webp']`.

**Correction appliquée** : Ajout de `file_size_limit: 5242880` et `allowed_mime_types: ARRAY['image/jpeg', 'image/png', 'image/webp']` dans le `INSERT INTO storage.buckets`.

**Statut** : CORRIGÉ

## 5. Warnings (non bloquants)

### [WARNING] W1 — Couleurs hardcodées : gradient vert terrain `#1a472a` / `#2d6a4f`

**Lignes** : 196, 378

**Valeur** : `'linear-gradient(135deg, #1a472a 0%, #2d6a4f 100%)'`

**Analyse** : Ces couleurs sont spécifiques au rendu fallback "terrain de football" — elles n'ont pas d'équivalent dans les tokens actuels (`@aureak/theme`). Le `as any` est justifié (CSS web non supporté nativement par StyleSheet). Acceptable en l'absence d'un token `colors.terrain.*`, mais à créer dans une story de polish.

**Statut** : WARNING — non bloquant, documenté.

---

### [WARNING] W2 — `#FFFFFF` hardcodé sur éléments de header

**Lignes** : 203, 390, 396, 406

**Valeur** : `color: '#FFFFFF'` sur textes overlay header (bouton retour, badge joueurs, nom implantation)

**Analyse** : `colors.light.surface` = `'#FFFFFF'` mais sémantiquement `surface` n'est pas "texte blanc sur fond sombre". Un token `colors.text.onDark` ou `colors.text.white` serait plus expressif. Pour l'instant, l'usage de `'#FFFFFF'` littéral est toléré sur les overlays sombres (pattern établi dans d'autres stories).

**Statut** : WARNING — non bloquant, à adresser si token `colors.text.onDark` est créé.

---

### [WARNING] W3 — `rgba(0,0,0,...)` hardcodé pour overlays

**Lignes** : 212, 385, 963, 973, 1086

**Valeur** : `rgba(0,0,0,0.35)`, `rgba(0,0,0,0.45)`, `rgba(0,0,0,0.7)`

**Analyse** : Overlays de transparence sur photos — pas d'équivalent token. Pattern identique à d'autres pages (stages, clubs). Toléré comme convention établie pour les overlays photo.

**Statut** : WARNING — non bloquant.
