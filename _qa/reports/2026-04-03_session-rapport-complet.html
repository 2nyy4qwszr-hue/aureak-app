story-12-1 + 12-2
Modèle données gamification + Event bus
commit 63ad3a4
✅ DONE
Objectif
Créer le modèle de données complet pour récompenser les joueurs : badges gagnables, ledger de points immuable, items cosmétiques avatar débloquables. RPC centrale award_badge_if_applicable() gérant 4 événements.
Avant
Aucune table de gamification dans le répertoire actif supabase/migrations/. Les types TypeScript BadgeDefinition, PlayerBadge, PlayerPointsLedger n'existaient pas. L'app n'avait aucun mécanisme de récompense pour les enfants.
Comment c'a été fait
Migration 00106 créée avec 9 tables (badge_definitions, player_badges, player_points_ledger, avatar_items, player_avatars, player_unlocked_items, player_theme_mastery, skill_cards, player_skill_cards). Trigger fn_badge_awarded_ledger : à chaque INSERT badge → points automatiquement ajoutés au ledger. RPC SECURITY DEFINER award_badge_if_applicable avec idempotency via processed_operations. 12 types TypeScript ajoutés dans entities.ts.
Maintenant
Le système peut récompenser un enfant pour : acquérir un thème (QUIZ_MASTERED), revalider une compétence (SKILL_MASTERED), ou une attribution coach (COACH_AWARD). Le ledger est append-only — aucune suppression ou modification de points possible, même pour l'admin.

story-12-4
Quêtes hebdomadaires — attribution & progression
commit 63ad3a4
✅ DONE
Objectif
Système de quêtes hebdomadaires automatiquement assignées aux enfants (3 types : assister à des séances, acquérir des thèmes, compléter des révisions). Progression en temps réel lors de chaque événement.
Avant
Aucune mécanique de quête. Les enfants n'avaient aucun objectif hebdomadaire concret pour maintenir leur engagement. L'event bus (12.2) ne gérait pas la progression des quêtes.
Comment c'a été fait
Migration 00107 : tables quest_definitions (catalogue) + player_quests (instances par enfant par période). Types TypeScript : QuestDefinition, PlayerQuest, QuestStatus, QuestType.
Maintenant
Un enfant peut recevoir des quêtes hebdomadaires. La progression se met à jour automatiquement à chaque quiz ou séance. Quêtes expirées automatiquement en fin de période. Contrainte UNIQUE(child_id, quest_id, period_start) garantit l'idempotency.

Pistes d'amélioration
3 axes prioritaires pour aller plus loin
PRIORITÉ HAUTE

🧪 Couverture de tests insuffisante sur les nouvelles features

Les 34 stories implémentées en batch ont toutes un Gate 1 (lint, try/finally, console guards) mais très peu de tests unitaires ou E2E ciblés sur les nouvelles fonctionnalités. Le pipeline CI tourne (Vitest + RNTL + Playwright smoke), mais il ne couvre pas les nouveaux composants comme ErrorBoundary, useToast, usePersistedFilters, le calendrier custom, ou l'import CSV.

Ce qu'on recommande :

  • Ajouter un test Vitest pour chaque hook créé (useToast, usePersistedFilters) — logique pure, facile à tester
  • Ajouter des tests RNTL pour ErrorBoundary (simuler un crash et vérifier le fallback)
  • Ajouter un test Playwright E2E qui vérifie le flow import CSV (upload fichier → preview → confirmation → count joueurs)
  • Objectif : 60%+ de couverture sur les nouveaux fichiers dans components/ et hooks/
PRIORITÉ MOYENNE

🏗️ Architecture des notifications — découplage du sidebar

Le système de notifications in-app (tbd-notifs-inapp) est fonctionnel mais le badge et la logique de polling sont directement dans _layout.tsx. À mesure que l'app grandit, ce layout accumule des responsabilités : notifications, recherche globale, error boundary, toasts. Risque de "God component" difficile à maintenir.

Ce qu'on recommande :

  • Créer un NotificationProvider (React Context) qui encapsule le state des notifs, le polling et le markAsRead
  • Créer un SearchProvider pour isoler la recherche globale
  • Le _layout.tsx ne devrait que consommer ces contexts, pas les implémenter
  • Bonus : avec un Provider centralisé, on peut brancher Supabase Realtime pour des notifs push vraiment temps réel (pas de polling)

🗄️ Réconciliation des deux répertoires de migrations

La divergence entre aureak/supabase/migrations/ (00001–00090) et supabase/migrations/ (00096–00110) crée une ambiguïté persistante : quel répertoire est la source de vérité ? Quelle migration s'applique en premier ? La migration 00108 créée aujourd'hui documente le problème mais ne le résout pas structurellement.

Ce qu'on recommande :

  • Créer une migration "méga-baseline" qui inclut en mode IF NOT EXISTS toutes les tables des migrations historiques (00001–00090 d'aureak/). Ce fichier devient le point de départ officiel pour un nouveau Supabase project
  • Archiver aureak/supabase/migrations/ dans un dossier _archive/ pour éviter la confusion
  • Configurer le supabase/config.toml racine comme source unique : migrations_dir = "supabase/migrations"
  • Mettre à jour CLAUDE.md pour clarifier que supabase/migrations/ (racine) est la seule source de vérité
