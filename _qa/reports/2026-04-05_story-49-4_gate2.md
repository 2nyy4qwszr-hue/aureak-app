## Bug Hunter — Edge Cases

### [WARNING] — Double-tap dans la fenêtre d'erreur (timing race)

**Fichier** : `page.tsx:315-350`
**Description** : Scénario : L'appel API échoue → rollback → `attendanceError` affiché pendant 4s. L'utilisateur clique à nouveau pendant ce délai. Le `attendanceToggling.has(childId)` est `false` (finally a retiré l'id), donc le second clic passe. C'est le comportement correct (réessayer est souhaité), mais `setAttendanceError(null)` est appelé au début du handler (ligne 326), ce qui efface le message d'erreur précédent immédiatement. Le `setTimeout` précédent de 4s continue de tourner et appellera `setAttendanceError(null)` après 4s, ce qui n'a pas d'impact visible mais laisse un timer orphelin.
**Impact** : Timer setTimeout orphelin par retry. Accumulation mémoire négligeable sur des interactions normales. Non bloquant.
**Correction suggérée** : Stocker le ref du timer et le clearTimeout avant d'en créer un nouveau.

---

### [WARNING] — Rollback non visible si composant démonté pendant l'appel

**Fichier** : `page.tsx:327-349`
**Description** : Si l'utilisateur navigue hors de la page pendant un appel `recordAttendance` en cours, le `setAttendanceMap` du rollback sera appelé sur un composant démonté. React 18 ne lance plus d'erreur mais le setState est silencieusement ignoré.
**Impact** : Aucun crash, mais la présence peut être mal persistée sans feedback. Peu probable en usage normal.
**Note** : Pattern commun, acceptable sans `useRef` isMounted guard pour cette story.

---

### [WARNING] — Séance sans groupId : `attendanceMap` non utilisé mais initialisé vide

**Fichier** : `page.tsx:150-153`
**Description** : Pour une séance ponctuelle (`groupId = null`), la boucle `sortedMembers.forEach(m => { map[m.childId] = null })` ne s'exécute pas (`sortedMembers = []`), puis `a.data.forEach(att => { map[att.childId] = att.status })` peuple quand même la map avec les présences existantes. Cette map n'est jamais lue dans la section Présences ponctuelle (qui lit `attendances[]` directement), mais reste en mémoire.
**Impact** : Redondance mémoire légère. `attendanceMap` n'est pas utilisé pour le rendu ponctuel — AC7 respecté. Non bloquant.

---

### [INFO] — Compteur présents exclu les statuts avancés

**Fichier** : `page.tsx:813`
**Description** : `groupMembers.filter(m => attendanceMap[m.childId] === 'present')` ne compte que les `present` stricts. Un joueur avec statut `late` (en retard mais présent) n'est pas compté. La spec story ne précise pas ce cas.
**Impact** : Le compteur peut sous-estimer la présence réelle. Acceptable pour V1 — les statuts avancés sont gérés dans `/presences`.

---

### [INFO] — Statut `'excused'` mentionné dans la story mais absent du code et de l'enum

**Fichier** : story-49-4.md (documentation)
**Description** : La story (contexte technique) mentionne `AttendanceStatus = 'present' | 'absent' | 'late' | 'trial' | 'injured' | 'excused'`. En réalité, l'enum DB (`00002_create_enums.sql`) n'a pas `'excused'`, et le type TS dans `@aureak/types/enums.ts` a `'unconfirmed'` à la place. Le code implémenté utilise `'unconfirmed'` (conforme au type TS). Incohérence dans la documentation de story uniquement.
**Impact** : Aucun impact runtime. À clarifier dans la story pour éviter confusion future.
