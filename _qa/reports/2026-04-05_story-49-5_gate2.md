### [WARNING] W5 — SparklineSVG : valeurs `undefined` ou `NaN` dans le tableau

**Fichier** : `page.tsx:231-271`
**Description** : `SPARKLINE_SEANCES = [8, 10, 9, 12, 11, totalSessions]` — si `visibleStats` est vide
(filtre implantation sans séances), `totalSessions = 0`. La sparkline contiendra `[8, 10, 9, 12, 11, 0]`
ce qui est valide mais trompeusement décroissant. Cas plus critique : si `totalSessions` vaut `NaN`
(multiplication sur `undefined`), `Math.min/max` retournerait `NaN` et tous les points seraient `NaN,NaN`,
rendant des coordonnées SVG invalides (polyline invisible).
Actuellement `totalSessions = visibleStats.reduce((s, i) => s + (i.sessions_total ?? 0), 0)` — le
`?? 0` protège contre `undefined` mais pas contre un type incorrect en entrée.
**Probabilité** : faible (données bien typées par Supabase), mais possible en edge case réseau.
**Correction suggérée** : Ajouter guard dans SparklineSVG :
```tsx
const safeValues = values.filter(v => typeof v === 'number' && !isNaN(v))
if (safeValues.length < 2) return null
```

### [WARNING] W7 — NextSessionTile : label "EN RETARD" même si séances viennent d'être créées (porté depuis Gate 1 W4)

**Fichier** : `page.tsx:536`
**Description** : Toute séance non clôturée affiche "EN RETARD" avec fond rouge. Une séance créée
aujourd'hui et non encore clôturée est considérée "EN RETARD" même si elle se déroule dans 2 heures.
La story AC4 prévoit une distinction "OUVERTE" / "EN RETARD" selon une logique temporelle. L'impl
actuelle simplifie à "EN RETARD" systématique.
**Impact UX** : Faux positif alarmant pour les admins. Séances planifiées futures = alerte rouge inutile.
**Action** : A corriger dans une prochaine story (calcul timestamp séance vs `now()`).

### [INFO] I1 — Sparkline simulée : `childrenTotal` peut être `null` lors du premier rendu

**Fichier** : `page.tsx:723`
**Description** : `SPARKLINE_JOUEURS = [42, 45, 41, 48, 47, childrenTotal ?? 50]`
Si `loadingCounts` est encore `true` et `childrenTotal === null`, la dernière valeur = `50` (fallback).
La sparkline s'affichera avec `50` comme dernier point même si la vraie valeur est différente.
Elle se met à jour lors du prochain rechargement mais pas en temps réel si `loadingCounts` se termine
après le rendu initial. Non-bloquant car les données sont simulées (story explicite).

---

### [INFO] I2 — Position NextSessionTile après QuickActions (légère déviation spec)

**Fichier** : `page.tsx:959-980`
**Description** : La story AC4 précise "insérée entre les KPI bento et le panneau anomalies".
L'implémentation place la tile APRÈS la barre Quick Actions (elle-même entre bento et anomalies).
Séquence actuelle : Bento → Quick Actions → NextSessionTile → Anomalies.
Séquence spec : Bento → NextSessionTile → Anomalies.
La Quick Actions bar est un composant pré-existant non mentionné dans la story, ce qui crée
l'ambiguité. Non-bloquant.

---

### [INFO] I3 — `SPARKLINE_SEANCES` contient `totalSessions` non-filtré par implantation dans certains cas

**Fichier** : `page.tsx:724`
**Description** : `totalSessions` est calculé depuis `visibleStats` (filtré par implantation). La
sparkline historique simule des données globales `[8, 10, 9, 12, 11, totalSessions]`. Si l'admin
filtre par une implantation avec 2 séances, la sparkline affiche `[8, 10, 9, 12, 11, 2]` — le
dernier point en rupture est correct (il reflète le filtre actif) mais les 5 points précédents
sont toujours des données globales simulées. Incohérence documentée jusqu'à l'API historique.
