## 3. Security Auditor

### RLS

| Table | RLS activé? | Politiques après migration |
|-------|-------------|---------------------------|
| `tickets` | OUI (00092) | `tickets_parent_own` (recréée) + `tickets_staff_all` (inchangée) |

### Analyse de la policy recréée

**Ancienne policy (00092)** :
```sql
CREATE POLICY "tickets_parent_own" ON tickets
  FOR ALL USING (
    parent_id = auth.uid()
    AND tenant_id = current_tenant_id()
    AND is_active_user()
  );
```

**Nouvelle policy (00144)** :
```sql
CREATE POLICY "tickets_parent_own" ON tickets
  FOR ALL USING (
    deleted_at IS NULL
    AND (parent_id = auth.uid() OR current_user_role() IN ('admin','coach'))
  );
```

#### [BLOCKER] Policy `tickets_parent_own` — régression sécurité

La nouvelle policy supprime deux contrôles présents dans 00092 :
1. `tenant_id = current_tenant_id()` → **supprimé** → un parent peut potentiellement voir les tickets d'autres tenants si son `auth.uid()` correspond à un `parent_id` cross-tenant
2. `is_active_user()` → **supprimé** → un utilisateur désactivé peut accéder à ses tickets
3. La condition `OR current_user_role() IN ('admin','coach')` est ajoutée dans la policy **parent** — cela duplique partiellement la coverage de `tickets_staff_all` mais de façon incomplète (manque `tenant_id` check)

Ce BLOCKER est **corrigible** : la policy doit être mise à jour pour réintroduire `tenant_id = current_tenant_id()` et `is_active_user()`.

**Correction recommandée** :
```sql
DROP POLICY IF EXISTS "tickets_parent_own" ON tickets;
CREATE POLICY "tickets_parent_own" ON tickets
  FOR ALL
  USING (
    deleted_at IS NULL
    AND parent_id = auth.uid()
    AND tenant_id = current_tenant_id()
    AND is_active_user()
  );
```
Note : les admins/coaches sont couverts par `tickets_staff_all` — ne pas les inclure dans `tickets_parent_own`.
