## Migration 00118 — Analyse

### Numérotation
La migration 00118 suit 00117 (`implantations_photo_url.sql`). Séquence correcte.
Note : un gap pré-existant 00114/00115 est présent dans le dépôt (non causé par cette story).

### Idempotence
`CREATE OR REPLACE VIEW` — idempotente. ✅

### Sécurité / RLS
La vue `v_child_current_club` est une vue PostgreSQL standard (SECURITY INVOKER par défaut).
Les tables source `child_directory_history` et `academy_seasons` ont toutes deux RLS activé
(migrations 00046 et 00041). L'isolation tenant est donc assurée via le JWT Supabase transmis
par `@aureak/api-client`. Conformité avec le pattern établi par `v_club_gardien_stats` (00113).

### SQL — Logique métier
- `DISTINCT ON (tenant_id) … ORDER BY tenant_id, start_date DESC` : protection contre plusieurs
  saisons `is_current = true` — conforme AC1.
- Join `es.saison_label = h.saison` : suppose que `academy_seasons.label` est au format identique
  à `child_directory_history.saison` (ex: '2025-2026'). Documenté en Technical Notes.
- `LEFT JOIN club_directory` : correct, `club_directory_id` est nullable.
- Aucune donnée sensible enfant (pas de nom, email, téléphone, adresse exposés dans la vue).

### Synchronisation DB ↔ TypeScript
