### 1.4 Dépendance critique : `attendance_started_at` sur `sessions`

**WARNING** : Le trigger `trg_attendance_start_session` (ligne 102) fait référence à la colonne `sessions.attendance_started_at`. Cette colonne est définie dans l'archive (`00014_sessions.sql` archive) mais **aucune migration active dans `supabase/migrations/` ne la crée explicitement**.

- Grep exhaustif sur `supabase/migrations/*.sql` → aucun `ADD COLUMN … attendance_started_at`
- Elle n'est référencée que dans `00136` (le trigger)
- Si `sessions.attendance_started_at` n'existe pas en remote, le trigger échouera à l'exécution

**BLOCKER potentiel à vérifier** : si la colonne est absente de `sessions` en remote, chaque INSERT dans `attendances` provoquera une erreur SQL dans le trigger. La migration s'appliquera sans erreur (la création de la fonction ne vérifie pas l'existence de la colonne), mais le trigger échouera au runtime.

**Recommandation** : Ajouter en début de migration :
```sql
ALTER TABLE sessions
  ADD COLUMN IF NOT EXISTS attendance_started_at TIMESTAMPTZ;
```

### 1.5 Dépendance : `block_checkins` — FK manquante

**WARNING** : Dans l'archive originale (00016), `block_checkins.session_block_id` référençait `session_blocks(id)`. Dans 00136, la FK est intentionnellement supprimée avec le commentaire :
> "Note : session_block_id est un UUID libre (pas de FK vers session_blocks car cette table n'est pas encore créée dans les migrations actives)"

Ce choix est documenté et défendable. Il introduit cependant un risque d'orphelins. À surveiller lors de la création future de `session_blocks`.

### 2.2 Isolation tenant

| Check | Résultat |
|-------|----------|
| Toutes les policies admin vérifient `tenant_id = current_tenant_id()` | PASS |
| Policies coach vérifient `tenant_id = current_tenant_id()` | PASS |
| Policies parent **ne vérifient pas** `tenant_id` directement | NOTE |
| Policy club_read **ne vérifie pas** `tenant_id` directement | NOTE |

**Note sur parent/club** : Les policies `att_parent_read` et `sa_parent_read` filtrent via `parent_child_links` (lien direct enfant ↔ parent) sans clause `tenant_id`. Cela est acceptable car le lien `parent_child_links.parent_id = auth.uid()` est déjà une isolation par utilisateur — un parent ne peut voir que ses propres enfants. Idem pour `att_club_read` via `club_child_links`. Comportement correct et attendu.

### 2.3 Principe du moindre privilège

| Rôle | Table | Droits accordés | Évaluation |
|------|-------|-----------------|------------|
| Admin | Toutes | ALL (lecture + écriture) | Correct (rôle superviseur) |
| Coach | `session_attendees` | SELECT (coach_read) + ALL si coach de la séance (coach_write) | PASS — double policy, granularité correcte |
| Coach | `attendances` | ALL si coach de la séance via `session_coaches` | PASS |
| Coach | `coach_presence_confirmations` | ALL sur ses propres lignes (`coach_id = auth.uid()`) | PASS |
| Coach | `block_checkins` | ALL sur ses propres check-ins | PASS |
| Parent | `session_attendees` | SELECT uniquement via lien enfant | PASS |
| Parent | `attendances` | SELECT uniquement via lien enfant | PASS |
| Enfant | `attendances` | SELECT uniquement `child_id = auth.uid()` | PASS |
| Club | `attendances` | SELECT uniquement via `club_child_links` | PASS |
| Club | `session_attendees` | **Aucune policy** | NOTE |

**Note club / session_attendees** : Le rôle `club` n'a pas de policy sur `session_attendees`. Si un partenaire club tente d'accéder au roster prévu d'une séance, RLS bloquera par défaut (deny-by-default). C'est un choix de conception acceptable — le roster reste visible uniquement aux admins, coachs et parents.

### 2.4 `SECURITY DEFINER` — Analyse des fonctions

**`trg_attendance_start_session`** (trigger, SECURITY DEFINER) :
- SET search_path = public : PASS (protection injection)
- Opération : UPDATE sessions WHERE id = NEW.session_id — aucune fuite de données, opération bornée

**`prefill_session_attendees`** (RPC, SECURITY DEFINER) :
- SET search_path = public : PASS
- Vérifie `tenant_id = current_tenant_id()` dans le SELECT sessions → PASS
- INSERT depuis `group_members WHERE group_id = v_group_id` — pas de vérification que v_group_id appartient bien au tenant
- **WARNING mineur** : si `current_tenant_id()` retourne NULL (JWT mal formé), la clause `WHERE id = p_session_id AND tenant_id = current_tenant_id()` ne matchera rien → `v_group_id` sera NULL → INSERT sur `group_members WHERE group_id IS NULL` → 0 résultats → retourne 0 sans erreur. Comportement silencieux mais non dangereux (pas de fuite cross-tenant).

## 6. Verdict Final

```
VERDICT : FAIL (BLOCKER B-01)
```

La migration est structurellement solide, idiomatique, et respecte l'ensemble des AC de la story. La couverture RLS est complète et appropriée pour des données de présences de mineurs. L'idempotence est garantie sur l'ensemble des DDL.

**Cependant**, le BLOCKER B-01 est critique : le trigger `trg_attendance_start_session` référence `sessions.attendance_started_at`, colonne absente des migrations actives. Ce bug silencieux au déploiement rendra toute tentative d'enregistrement de présences non fonctionnelle (erreur SQL au runtime sur chaque INSERT dans `attendances`), ce qui inverse l'objectif P0 de la story.

### Correction minimale requise

Ajouter en tête de section (avant le trigger, après la création des tables) :

```sql
-- Colonne attendance_started_at sur sessions (définie dans archive 00014 non migrée en actif)
ALTER TABLE sessions
  ADD COLUMN IF NOT EXISTS attendance_started_at TIMESTAMPTZ;
```

Après correction, re-soumettre en Gate 1 → la migration devrait passer PASS.
