## Agent Bug Hunter (statique)

### STOP — 5 points prioritaires
1. **`.data.field` sans optional chaining** : ligne 78 `gradesResults[i].data` — safe (index `i` de `Promise.all` même longueur que `allRows`, type retour garanti `CoachGrade | null`)
2. **Double-soumission** : N/A — page lecture seule (pas de formulaire submit)
3. **try/catch silencieux** : ✅ catch loggue avec guard NODE_ENV
4. **Erreur Supabase ignorée** : ⚠️ `listCoaches` — `error` non déstructuré à la ligne 68 (`const { data: allRows }`) — si Supabase retourne une erreur, `allRows` sera `[]` (le wrapper API gère déjà ce cas), donc non-BLOCKER
5. **Mutation sans invalidation** : N/A — page lecture seule

### Edge cases
- `roleFilter === 'coach'` → filtre `() => roleFilter !== 'assistant'` retourne TOUS les coaches, pas seulement les coachs (commenté dans le code comme limitation connue future). Comportement documenté → WARNING non-bloquant.
- `displayStart = page * PAGE_SIZE + 1` quand `filtered.length === 0` → affiche "Affichage de 1–0 / 0" — mais ce cas est géré avant par `filtered.length === 0 → "Aucun entraîneur"` dans le JSX, donc non-affichable.

### Régressions
- Story purement UI, aucune API modifiée, aucune migration. Zéro risque de régression.

## Warnings (non-bloquants)
1. **Pas d'état d'erreur UI** : si `listCoaches` échoue (ex: réseau), la page affiche "Chargement…" puis rien (loading→false, coaches=[]→"Aucun entraîneur"). L'utilisateur ne sait pas pourquoi la liste est vide. Suggestion : ajouter un `[error, setError]` state avec message.
2. **Filtre COACH** : `roleFilter === 'coach'` ne filtre pas réellement (tous les coaches retournés). Le commentaire dans le code documente clairement la limitation future. Acceptable pour cette story.
3. **Hover state table** : `rowPressed: { opacity: 0.75 }` uniquement sur press — pas de hover desktop. Pattern cohérent avec le reste du codebase.
