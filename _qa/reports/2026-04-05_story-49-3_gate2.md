### Edge case 1 — Club sans joueur annuaire

**Scénario** : `listChildrenByClubDirectoryId(clubId)` retourne `{ data: [], error: null }`.

**Analyse** : `setAnnuairePlayers([])` → `annuairePlayersDeduped` = `[]` → la branche `annuairePlayersDeduped.length === 0` affiche "Aucun joueur lié via l'annuaire." (AC5). PASS.

**Scénario** : `listChildrenByClubDirectoryId(clubId)` retourne `{ data: [], error: <erreur Supabase> }`.

**Analyse** : La function retourne `{ data: [], error }` sur erreur — donc `annuaireRes.data = []`. `setAnnuairePlayers([])` → section vide. L'erreur Supabase est silencieuse côté UI pour cette sous-requête (le `catch` du `Promise.all` au niveau de `load()` ne capture que les exceptions `throw` — et `listChildrenByClubDirectoryId` ne throw pas, elle retourne `error`).

**[WARNING-4]** : `page.tsx:438` — Si `listChildrenByClubDirectoryId` retourne une erreur Supabase, elle est absorbée silencieusement (`annuaireRes.error` n'est pas vérifié). L'utilisateur voit une section vide sans feedback. Les autres fonctions du `Promise.all` (`listChildrenOfClub`, `listCoachesOfClub`) ont le même comportement (pre-existant) — il n'y a pas de vérification de `.error` sur les résultats intermédiaires. Cohérent avec le pattern existant. Non bloquant mais à améliorer globalement.

### Edge case 2 — Déduplication si même joueur dans les 3 listes

**Scénario** : Un joueur `id = "X"` est dans :
- `club_directory_child_links` avec `link_type = 'current'` → `currentPlayers` contient `{ childId: "X", ... }`
- `club_directory.club_directory_id = clubId` → `annuairePlayers` contient `{ id: "X", ... }`

**Analyse** :
```typescript
const currentIds    = useMemo(() => new Set(currentPlayers.map(p => p.childId)), [...])
const affiliatedIds = useMemo(() => new Set(affiliatedPlayers.map(p => p.childId)), [...])
const linkedIds = new Set([...currentIds, ...affiliatedIds])
return annuairePlayers.filter(p => !linkedIds.has(p.id))
```

`currentIds` utilise `.childId`, `annuairePlayers` utilise `.id` — les deux sont le `child_directory.id` UUID. La déduplication est correcte. PASS.

**Scénario** : Un joueur est dans `affiliated` ET dans l'annuaire.

**Analyse** : `affiliatedIds` contient `childId`, et `linkedIds` inclut `affiliatedIds` — le filtre exclut le joueur. PASS.

**Scénario** : Un joueur est dans `current`, `affiliated` ET dans l'annuaire.

**Analyse** : `linkedIds = new Set([...currentIds, ...affiliatedIds])` — la déduplication par Set est idempotente. PASS.

**Scénario** : Un joueur est UNIQUEMENT dans l'annuaire (ni current ni affiliated).

**Analyse** : `linkedIds` ne contient pas son ID → `annuairePlayersDeduped` l'inclut → affiché dans la section annuaire. PASS. C'est le cas principal visé par la story.

### Edge case 3 — `clubId` undefined (navigation directe vers route sans paramètre)

**Analyse** : `const { clubId } = useLocalSearchParams<{ clubId: string }>()` — en cas de navigation anormale, `clubId` peut être `undefined`. La garde `if (!clubId) return` (ligne 412 dans `load()`) protège correctement. `listChildrenByClubDirectoryId(undefined)` ne serait jamais appelée. PASS.

### Edge case 4 — Volume (> 100 joueurs annuaire)

**Analyse** : `listChildrenByClubDirectoryId` n'a pas de pagination. La story note explicitement : "Si un club dépasse 100 joueurs annuaire, ajouter une pagination dans une story dédiée." Pour les clubs belges de football jeunes, le volume est inférieur à 100. Acceptable pour cette story.

**[INFO]** : Sans pagination, la section annuaire peut devenir lente pour de grands clubs. À surveiller.

### Edge case 5 — Rafraîchissement pendant chargement

**Analyse** : Si `useEffect` déclenche `load()` et le composant est démonté pendant le `Promise.all`, tous les `setState` appelés dans le `try` après le `await` mettront à jour un composant démonté. Il n'y a pas de cleanup (AbortController). Ce pattern est pre-existant dans le fichier, non introduit par cette story. Non bloquant ici.

### Edge case 6 — `listChildDirectory` avec `pageSize: 500`

**Analyse** : `listChildDirectory({ page: 0, pageSize: 500, actif: true })` est pre-existant (chargement du picker). `listChildrenByClubDirectoryId` est une requête séparée sur `club_directory` avec filtre `club_directory_id`. Les deux requêtes sont indépendantes — pas d'interférence. PASS.
