## Ce qui a marché

### Factorisation anticipée

Le composant `PeopleListPage` créé en 87.1 a trivialisé 87.2 (fiche universelle) et 87.3 (onglet Accès). Les 3 pages concrètes (managers/commerciaux/marketeurs) sont chacune ≤ 40 lignes parce que toute la logique commune est mutualisée. ROI immédiat sur les stories suivantes.

### Modules role-aware autonomes

Le pattern "chaque module vérifie son rôle en interne et retourne `null` si non applicable" (87.2) évite tout `switch/case` central. Conséquence : ajouter un rôle futur = 1 fichier + 1 ligne d'import dans `ResumeTab.tsx`. Extensibilité gratuite et coût de maintenance marginal.

### Stub + TODO comme contrat d'interface

Le stub `/profiles/[userId]` posé en 87.1 et le placeholder `AccesTab.tsx` de 87.2 ont permis à chaque story d'être indépendamment déployable sans couplage serré. Cela a aussi rendu 87.3 transparent à implémenter (aucune modification de `ProfileTabs.tsx` ou `index.tsx`).

### Infrastructure Epic 86 réutilisée à 100%

- `section_permissions` + `user_section_overrides` + `profile_roles` consommées par 87.2 et 87.3 sans nouvelle colonne ni nouvelle table.
- Edge Function `create-user-profile` (Epic 10) compatible out-of-the-box avec commercial/manager/marketeur (validation pré-implémentation en 87.4 Task 5 a économisé un effort backend).
- `ROLE_LABELS` extrait dans `@aureak/ui/labels/roles.ts` (87.2) : source de vérité partagée, évite la dispersion.

### Auto-merge mid-epic

La décision d'activer `gh pr merge --auto --squash` par défaut (mid-session) a réduit le temps cumulé entre PR créée et PR mergée à quasi-zéro, tant que les checks verts passaient. Gain net sur les 3 dernières stories.

## Ce qui a posé problème

### Fragilité des PRs stackées

Le stacking 87.2 sur branche 87.1 a créé une confusion de merge. L'ordre effectif de clic était inversé (#57 puis #56), et GitHub a squashé les deux commits dans le merge de la PR #56, laissant la PR #57 dans un état où son contenu était déjà upstream. Le rebase automatique a donné un `patch already upstream` inattendu.

**Leçon** : éviter le stacking pour les prochaines stories dépendantes. Soit (a) attendre le merge du parent, soit (b) découper pour rendre la seconde story indépendante.

### Crédit Anthropic API épuisé → CI polluée

Le workflow `cto-review.yml` (7 agents review) et le health check `check-secret.yml` ont échoué sur toutes les PRs du début d'Epic 87, générant des checks rouges bruyants. Cause racine : crédit API à zéro (identique au handoff PR #20). La review sémantique IA est indisponible jusqu'à recharge.

**Mitigation appliquée** : PR #58 désactive le trigger PR sur ces deux workflows, en conservant `workflow_dispatch` pour relance manuelle.

### Playwright MCP locked toute la session

Le browser MCP Playwright a été verrouillé par une autre session parallèle tout au long de l'Epic. Zéro test E2E automatisé exécuté sur les 5 stories. QA réelle reposée uniquement sur `tsc --noEmit` + preview Vercel.

### Sprint-status drift

Le fichier `sprint-status.yaml` a dérivé plusieurs fois par rapport à la réalité des branches. Ex : 87.1 marquée `review` pendant l'implémentation de 87.2, puis passée à `done` uniquement dans la branche 87.3. Source potentielle de confusion pour un observateur extérieur ou une reprise de session.

### Garde admin dupliquée

Trois stories (87.2, 87.3, 87.4) utilisent chacune une variante légèrement différente de garde admin : `useAuthStore(s => s.role)` en simple, `getEffectivePermissions().admin`, `getEffectivePermissions().academie`. Toutes fonctionnelles mais documentent un gap d'unification identifié sans résolution programmée.

---

## Insights clés

1. **Factoriser tôt bat factoriser tard** — 100 lignes partagées en story 1 évitent 400+ lignes dupliquées sur les stories 2-3 du même épic.
2. **Relire l'Edge Function avant de l'utiliser** (87.4 Task 5) évite un refactor backend inutile — pattern à systématiser.
3. **Les PRs stackées sont contre-intuitives sur GitHub UI** — l'utilisateur clique dans l'ordre visible, pas dans l'ordre de dépendance. Risque d'artefacts de squash.
4. **Auto-merge + QA Gate 1 suffisent pour du code UI itératif** — la review IA sémantique serait un +, pas un must pour ce niveau.
5. **Redirect silencieux > suppression** pour déprécier une route (87.5) — préserve 100% des bookmarks sans overhead.
