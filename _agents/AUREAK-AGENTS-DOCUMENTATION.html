v5 — Changement majeur : Le Day Conductor orchestre mais ne code plus directement. La création de stories est déléguée au BMAD SM (contexte propre, workflow create-story, mode autonome) et l'implémentation au BMAD Dev (contexte propre, workflow dev-story, HALT conditions, tasks en ordre strict). Les Gates restent des agents Aureak custom car ils portent les règles spécifiques du projet (try/finally, console guards, 12 principes design, RLS Aureak) que BMAD générique ne connaît pas.

Agents Proactifs
7 agents — génèrent des propositions, pas des stories directement
PROACTIFS

Génération & Dispatch

Transforment une intention en story BMAD codable — le cœur du workflow quotidien

🌅
Morning Supervisor
Déclenché par : /morning
"Analyse l'état du projet et propose jusqu'à 10 stories priorisées pour la journée — Jeremy choisit les numéros, les stories se créent automatiquement."
Lit le backlog, les bugs connus, les pages existantes et le PRD. Présente un brief structuré en 4 catégories : BUGS / DESIGN / UX / FEATURES. Sur confirmation de Jeremy, déclenche la Story Factory pour chaque numéro choisi.
Input : /morning Output : propositions numérotées Fichier : .claude/commands/morning.md
⚗️
Story Factory
Déclenché par : story "décision" ou Morning
"Transforme une décision validée en langage naturel en fichier story BMAD complet, avec SQL, snippets de code et liste de fichiers — prêt à coder sans aucune question."
Lit l'architecture, le PRD, les migrations, les types et les stories existantes avant d'écrire. Génère le fichier dans _bmad-output/implementation-artifacts/ et met à jour BACKLOG.md automatiquement.
Input : décision 1-3 phrases Output : story BMAD .md Script : story.sh
📥
Feedback Dispatcher
Déclenché par : texte brut de Jeremy
"Analyse un retour brut en français, le catégorise en tickets BUG/DESIGN/UX/FEATURE avec priorité, vérifie les doublons existants, et déclenche la Story Factory sur confirmation."
Prend le retour d'expérience, une observation ou une frustration de Jeremy. Identifie chaque problème distinct, lit les fichiers concernés pour confirmer, et présente des tickets structurés avant d'agir.
Input : texte libre FR Output : tickets BUG/DESIGN/UX/FEAT Fichier : feedback-dispatcher.md
🎨
Design Patrol
Déclenché par : audit autonome périodique
"Parcourt toutes les pages de l'app via Playwright, détecte chaque dérive visuelle par rapport aux 12 principes design, et produit un rapport de dérives priorisées pour le Morning Brief."
Vérifie la palette (couleurs hardcodées ?), les anti-patterns (glassmorphism, dégradés violets, bento hors dashboard), les tokens manquants, et les états vides sans illustration. Chaque BLOCKER = story obligatoire.
Input : aucun Output : _qa/reports/design-patrol.md Fichier : design-patrol.md
🕷️
Bug Crawler
Déclenché par : audit autonome périodique
"Navigue dans l'app en simulant les interactions réelles, capture toutes les erreurs console JS et les crashes de rendu, et produit un rapport de bugs classés CRITICAL / HIGH / MEDIUM."
Injecte un collecteur d'erreurs global, crawle chaque route, teste les interactions clés (recherche, formulaires, navigation), et identifie les spinners infinis, les 404, et les promise rejections silencieuses.
Input : aucun Output : _qa/reports/bug-crawler.md Fichier : bug-crawler.md
⚡
UX Inspector
Déclenché par : audit autonome périodique
"Simule les 5 parcours réels d'un admin, compte le nombre de clics par action, identifie les états manquants et les incohérences de navigation, et propose des simplifications concrètes."
Mesure : créer une séance, trouver un joueur, ajouter un joueur à un club, consulter un planning stage, naviguer entre sections. Chaque friction > 3 clics est signalée avec une proposition de simplification.
Input : aucun Output : _qa/reports/ux-inspector.md Fichier : ux-inspector.md
🔭
Feature Scout
Déclenché par : revue de backlog périodique
"Compare le PRD à l'état réel du backlog, identifie les FRs Phase 1 non implémentés et les quick wins, et produit les 'Story Factory calls' prêts à copier-coller."
Construit une matrice de couverture PRD vs stories done/in-progress. Signale les FRs Phase 1 manquants (priorité absolue), les FRs Phase 2 débloqués, et les améliorations hors PRD à haute valeur.
Input : aucun Output : _qa/reports/feature-scout.md Fichier : feature-scout.md

Agents Réactifs — Gate 1 & Gate 2
7 agents — déclenchés après chaque implémentation de story
GATE 1

Pré-PR — Qualité du code

Toujours après implémentation, avant le commit. Lancer avec qa gate1 {story-id}

🔍
Code Reviewer
Gate 1 — toujours
"Analyse chaque fichier modifié de la story contre les règles absolues du projet : try/finally, console guards, accès Supabase via api-client, pas de couleurs hardcodées."
Vérifie les patterns interdits, les imports non conformes, et les violations des contraintes stack. Produit un rapport avec BLOCKER/WARNING/INFO et un verdict GO/NO-GO.
Input : story + fichiers modifiés Output : _qa/reports/code-review.md Fichier : code-reviewer.md
🗄️
Migration Validator
Gate 1 — si nouveau .sql
"Vérifie que chaque migration SQL est idempotente (IF NOT EXISTS), soft-delete conforme, RLS ajouté si nouvelle table, et synchronisée avec les types TypeScript dans entities.ts."
Lit toutes les migrations précédentes pour le contexte, compare le schéma DB aux types TS, vérifie la numérotation séquentielle et l'absence de migrations destructives (DROP, ALTER avec perte).
Input : fichier .sql + story Output : _qa/reports/migration-validator.md Fichier : migration-validator.md
🔐
Security Auditor
Gate 1 — si auth / RLS / données sensibles
"Audite les policies RLS, l'isolation multi-tenant, les Edge Functions, et l'exposition de données sensibles — uniquement sur les stories touchant l'auth ou les tables protégées."
Vérifie les JWT claims, les headers non-autoritaires, les fuites de tenant_id, l'exposition de PII, et les Edge Functions sans authentification. Déclenché uniquement quand c'est pertinent.
Input : migrations + edge functions Output : _qa/reports/security.md Fichier : security-auditor.md
GATE 2

Pré-Deploy — Qualité visuelle & intégration

Après Gate 1 validé. Lancer avec qa gate2 {story-id}

🖼️
Design Critic
Gate 2 — toujours si page .tsx
"Navigue dans l'app via Playwright sur la route de la story, vérifie les 12 principes design et les 6 anti-patterns BLOCKER, et produit un rapport visuel avec screenshots annotés."
Lit design-vision.md, prend des screenshots, vérifie la palette (tokens vs hardcodé), les ombres, les transitions, l'ambiance globale. Un anti-pattern = BLOCKER immédiat avant deploy.
Input : story + route + fichiers .tsx Output : _qa/reports/design-critic.md Fichier : design-critic.md
🧭
UX Auditor
Gate 2 — toujours si page .tsx
"Vérifie que les composants UI de la story respectent la spec UX : états vides, loading, erreur, feedback de succès, confirmations destructrices, cohérence des patterns d'interaction."
Consulte la UX design specification et les tokens de theme. Valide les états manquants, la conformité aux patterns établis, et l'absence de friction non justifiée dans le flux implémenté.
Input : story + fichiers .tsx Output : _qa/reports/ux.md Fichier : ux-auditor.md
🔗
Regression Detector
Gate 2 — si fichier partagé modifié
"Analyse les impacts en cascade quand un fichier partagé (entities.ts, api-client, tokens, _layout) est modifié, en listant toutes les stories done qui pourraient être affectées."
Déclenché uniquement si la story touche entities.ts, enums.ts, api-client/*.ts, tokens.ts, _layout.tsx ou une migration. Lit toutes les stories done pour identifier les dépendances cassées potentielles.
Input : fichiers partagés modifiés Output : _qa/reports/regression.md Fichier : regression-detector.md
🐛
Bug Hunter
Gate 2 — toujours (synthèse finale)
"Synthèse finale avant deploy : lit tous les rapports Gate 1 et Gate 2 précédents, teste les ACs de la story via Playwright, et produit le verdict GO/NO-GO avec liste de blockers."
Lance en dernier. Consolide les BLOCKER des autres agents, teste au moins 3 ACs manuellement, et met à jour _qa/summary.md avec le verdict final de la story. Sans son GO, pas de deploy.
Input : story + tous les rapports précédents Output : _qa/reports/bugs.md + summary.md Fichier : bug-hunter.md

Agents BMAD — Fondation du Projet
Utilisés lors de la phase de conception — génèrent les artéfacts de planification
BMAD

Méthode BMAD — Build, Measure, Analyze, Design

Phases séquentielles : Analysis → Planning → Solutioning → Implementation

📋
Analyst BMAD
Phase 1 — Analysis
"Produit le Product Brief à partir des inputs de Jeremy — vision, cibles, contraintes, personas — la fondation de tout le projet."
Input : vision produit Output : product-brief.md
📐
PM BMAD
Phase 2 — Planning
"Transforme le Product Brief en PRD complet avec 109 FRs numérotés, 3 phases de déploiement, et la UX Design Specification."
Input : product-brief.md Output : prd.md + ux-design-specification.md
🏗️
Architect BMAD
Phase 3 — Solutioning
"Conçoit l'architecture technique complète — stack, monorepo, patterns, contraintes — et la documente dans architecture.md, source de vérité technique du projet."
Input : prd.md Output : architecture.md
📝
Story Writer BMAD
Phase 3 — Solutioning
"Décompose les epics en stories BMAD détaillées — 80+ fichiers done couvrant les Epics 1–33 (Phase 1 complète + stories additionnelles)."
Input : architecture.md + prd.md Output : 80+ stories .md
⚙️
Dev Agent BMAD
Phase 4 — Implementation
"Implémente chaque story BMAD — migration SQL, types TS, API client, UI — en suivant les tâches checklistées et les dev notes de la story."
Input : story ready-for-dev Output : code + story → done
✅
QA Agent BMAD
Phase 4 — après implémentation
"Valide chaque story implémentée en testant les ACs manuellement via Playwright et en vérifiant la conformité aux règles absolues du projet."
Input : story done + app running Output : verdict QA
Note : Pour les grandes epics ou refactors complexes, utiliser BMAD Story Writer pour une décomposition rigoureuse. Pour les features et corrections du quotidien, utiliser directement la Story Factory (plus rapide, même format BMAD).

Design Vision — 12 Principes
Source : _agents/design-vision.md — référence pour Design Patrol, Design Critic, Story Factory
01
Light Premium
Fond beige #F0EBE1, cards blanches. Jamais fond dark hors sidebar.
02
Gold Accent Only
Seul #C9A84C pour les accents. Aucun violet, bleu, ou vert primaire.
03
Typographie Montserrat
900 display, 700 titre, 500 body, 400 caption. Jamais system-ui en prod.
04
Hiérarchie d'Ombres
shadows.sm repos, shadows.md hover, shadows.lg modales. Jamais flat partout.
05
Transitions Fluides
transitions.fast (150ms) sur hover/focus. Jamais sans transition.
06
Sidebar Dark Fixe
#111111 avec stripe gold 2.5px en haut. Seul élément dark du layout.
07
Bento Dashboard Only
Grille bento uniquement sur le dashboard. Sous-pages = listes/fiches.
08
Photo Humaine
Chaque page importante a une photo de gardien. Pas de pages 100% UI.
09
Coins Raisonnables
Max radius.cardLg (24px). Jamais de médaillons ou bulles excessives.
10
États Complets
Loading (skeleton), empty state (illus + CTA), error, success — tous définis.
11
Tokens Only
Zéro couleur hardcodée dans les composants. @aureak/theme obligatoire.
12
Densité Calibrée
Assez d'air pour respirer, assez dense pour l'admin. Ni trop spacieux, ni étouffant.
6 Anti-Patterns BLOCKER
🚫 Bento hors dashboard — grille asymétrique sur sous-pages
🚫 Dégradés violet / bleu — interdit dans l'app admin
🚫 Glassmorphism — backdrop-filter sur n'importe quel composant
🚫 Fond dark hors sidebar — hors hero tile dashboard
🚫 Texte blanc sur fond clair — lisibilité obligatoire
🚫 Border-radius > 24px — pas de bulles ni médaillons

Seuls cas où Claude Code s'arrête : blocker non corrigeable automatiquement, dépendance manquante, ou story qui contredit l'architecture existante. Dans ces cas : description précise du problème + action attendue de Jeremy.
